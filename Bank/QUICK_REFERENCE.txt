━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 SMARTIFI: Single-Threaded Architecture
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Main Loop:
    select() → recv from NAPAS → process → send response → repeat

Throughput Per Instance:
    ✓ 30-50 TPS (light load)
    ✓ 200-500 ms latency

Scalability Strategy:
    Deploy multiple instances + HAProxy load balancer
    
        HAProxy (round-robin)
            ├─ Smartifi-01
            ├─ Smartifi-02
            └─ Smartifi-03
    
    Result: 3 instances × 30-50 TPS = 90-150 TPS

Shared State Management:
    ✓ Redis (session cache)
    ✓ ZooKeeper (coordination)
    ✓ Database (transaction log)

Key Parameters:
    • Connection timeout: 30 seconds
    • Transaction timeout: 500 ms
    • Max connections: Limited by OS file descriptors
    • Database pool: Usually 1-5 connections

Pros:
    ✓ Proven stable architecture
    ✓ Simple failure detection (one process = one instance)
    ✓ Straightforward to monitor
    ✓ Predictable performance

Cons:
    ✗ Requires external load balancer
    ✗ More infrastructure (3+ servers)
    ✗ Higher operational overhead
    ✗ More complex deployment (Redis, ZooKeeper setup)

Deploy When:
    • You have DevOps team (3+ people)
    • Time to deploy: 2-3 weeks available
    • Budget: $80K+ for infrastructure
    • Need modular, independently scalable system

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 ACQ: Multi-Process Master-Worker Architecture
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Process Model:
    Master (atmifp):
        ├─ Listen on port 9002
        ├─ Accept connections
        └─ Fork worker process for each connection
    
    Workers (atmifpxx - multiple):
        ├─ Handle one transaction each
        ├─ 5-10 MB memory per worker
        └─ Automatic cleanup when done

Throughput Per Instance:
    ✓ Single worker: 5-6 TPS
    ✓ 20 workers: 100-120 TPS
    ✓ 50 workers: 200-300 TPS
    ✓ Each worker = 5-6 TPS baseline

Worker Pool Sizing:
    Light traffic:    2-5 workers
    Medium traffic:   10-20 workers
    High traffic:     20-50 workers
    Very high:        50-100+ workers
    
    Rule of thumb: Workers = Peak TPS × 3 (to handle bursts)

State Coordination:
    ✓ Shared Memory (SHM) - parent/child communication
    ✓ Semaphores - process synchronization
    ✓ Unix sockets - FD passing
    ✓ Single database connection pool

Key Parameters:
    • Initial worker pool: 3-5
    • Max worker pool: 100-200
    • Worker lifecycle: 5-10 seconds per transaction
    • Memory per worker: 5-10 MB

Pros:
    ✓ Built-in scaling (automatic fork on demand)
    ✓ No external load balancer needed
    ✓ Better failure isolation (one worker down = one transaction fails)
    ✓ More efficient memory usage
    ✓ Simpler deployment (single binary)
    ✓ Production-ready architecture

Cons:
    ✗ Less familiar to some teams (fork-based)
    ✗ Process cleanup requires tuning (zombie processes)
    ✗ Requires understanding of SHM + semaphores

Deploy When:
    • Quick deployment needed (< 1 week)
    • Small-medium team
    • Single data center
    • Need production-ready system immediately
    • Budget: $30K-50K (simpler infrastructure)

 PERFORMANCE TUNING CHECKLIST
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

FOR SMARTIFI (increase TPS by 50%):
    1. ✓ Verify HAProxy load balancer is round-robin (not least-conn)
    2. ✓ Check TCP buffer sizes (sysctl -a | grep net.core)
    3. ✓ Tune database query performance (slow query log)
    4. ✓ Enable connection pooling in database
    5. ✓ Check Redis memory usage and eviction policy
    6. ✓ Review network latency (ping between instances)
    7. ✓ Monitor transaction timeout duration

FOR ACQ (increase TPS by 50%):
    1. ✓ Increase max worker processes (tune kernel limits)
    2. ✓ Verify database connections are pooled (not 1 per worker)
    3. ✓ Check kernel TCP backlog (net.core.somaxconn)
    4. ✓ Monitor shared memory (SHM) utilization
    5. ✓ Tune worker timeout (too short = thrashing, too long = resource waste)
    6. ✓ Review zombie process cleanup
    7. ✓ Monitor CPU context switching
